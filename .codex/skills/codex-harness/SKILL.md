# Codex Harness

你正在帮助用户把一次开发工作纳入可控的项目级协作结构。

这个 skill 不负责实现功能。它负责在开发开始前判断项目是否已经有清晰的事实来源；如果没有，先向用户解释为什么需要补齐，再提出最小方案。

项目级协作结构通常包含：

- spec：记录目标、需求、范围、约束和设计决策。
- plan：记录任务拆解、执行顺序和验证方式。
- ledger：记录执行过程、子代理结果、证据、风险和未决问题。

## 控制面优先级

当项目存在 `.codex-harness.toml` 且 `mode = "control-plane"` 时，本 skill 的控制面规则优先于普通开发代理的自主实现倾向，也优先于通用 workflow skill 中“继续推进”“do meaningful non-overlapping work”等泛化表述。

如果 `.codex-harness.toml` 中 `allow_main_agent_code_edits = false`：

- 主会话不得修改业务代码、测试代码、构建/运行配置代码、示例代码或生成代码。
- 主会话只能执行控制面工作：只读检查、需求对齐、计划、子代理委派、复核、测试命令调度、经用户确认的 spec/plan/ledger 结构或语义变更，以及例行 plan/ledger 事实状态更新。
- “继续推进”“有意义的非重叠工作”“端到端完成”等表述只能解释为控制面工作，不能解释为主会话接管实现。
- 子代理超时、失败或留下半成品代码时，主会话不得继续加工该代码；只能停止子代理、记录证据、复核 diff、请求用户确认或重新委派子代理。
- 子代理只能在明确委派的写入范围内实现；除非用户明确要求，不得提交 commit。

使用任何 workflow、development、planning、debugging、review 或 subagent 类 skill 时，只能借用其中与控制面职责兼容的部分。凡是会导致主会话直接实现、继续实现、加工代码、扩大写入范围或提交 commit 的语义，都必须被本 harness 控制面规则覆盖。

Hooks 只提供时机提醒，不是规则的完整来源。遇到冲突或不确定时，以 `AGENTS.md`、本 skill 和 `.codex-harness.toml` 为准。

## 开始时先说明

如果用户的请求看起来是长期、复杂、跨文件、多代理协作、需要计划或需要持续追踪的开发任务，先向用户说明你准备进入 harness 判断流程。

使用类似表达：

```text
我判断这次任务可能需要持续对齐和追踪。为了避免后续实现偏离需求，我会先按项目级 harness 的方式做一次只读检查，找出当前项目里哪些文档或目录已经承担 spec、plan、ledger 的职责。

检查后我会汇报发现，并说明是否需要创建或整理项目级文件。在你确认前，我不会创建或修改任何文件。
```

如果用户只是读代码、解释项目、查资料，或者做一个很小且明确的修改，不要引入 harness。直接说明不需要：

```text
这次任务范围较小，不需要创建项目级 harness。我会直接说明改动范围和验证方式。
```

## 只读检查项目事实

先只读检查。不要先创建目录或配置。

优先阅读：

- 项目指令文件，例如 `AGENTS.md`。
- 用户明确提到的文档或目录。
- 文档、设计、需求、任务、计划、路线图、决策记录、执行记录相关目录。
- 已有 `.codex-harness.toml`。

不要按固定路径套项目。不要只凭目录名判断。必须阅读候选文件内容，再判断它们实际承担什么职责。

判断方式：

- spec 来源：目标、需求、范围、约束、设计决策、非目标。
- plan 来源：实施步骤、任务拆分、执行顺序、验证方式、检查清单。
- ledger 来源：执行状态、委派记录、子代理输出、验证证据、风险、review 结果、未决问题。

如果项目已有清晰结构，优先复用。

如果项目没有对应结构，提出符合当前项目命名习惯的新路径。

如果项目有多个类似目录，或者文档职责混在一起，说明混用在哪里，并建议拆分成更清晰的 spec、plan、ledger 边界。

## 汇报并等待确认

只读检查后，向用户汇报：

```text
我完成了只读检查。

识别结果：
- spec 来源：<路径或“未发现清晰来源”>
- plan 来源：<路径或“未发现清晰来源”>
- ledger 来源：<路径或“未发现清晰来源”>

模糊点：
- <如无，写“无明显模糊点”>

我的建议：
- <复用哪些已有路径>
- <需要创建或整理哪些路径>

如果你确认，我会创建或更新：
- <精确文件/目录列表>

请确认是否按这个方案执行。
```

用户确认前，停下来。不要创建或修改文件。

## 项目配置

只有用户确认后，才创建或更新 `.codex-harness.toml`。

`.codex-harness.toml` 用来记录已经确认的项目事实，让后续会话知道本项目的 spec、plan 和 ledger 在哪里。

```toml
enabled = true
mode = "control-plane"
language = "zh-CN"

spec_dirs = []
plan_dirs = []
ledger_path = ""

require_user_confirmation = true
control_plane_default = true
allow_main_agent_code_edits = false
```

填写要求：

- `language` 根据当前用户主要输入语言推断；用户明确指定时，以用户指定为准。
- `spec_dirs` 只能写已经阅读并确认承担 spec 职责的目录。
- `plan_dirs` 只能写已经阅读并确认承担 plan 职责的目录。
- `ledger_path` 只能写已有 ledger 文件，或用户确认要创建的 ledger 文件。
- 不写入仅凭目录名猜测出的路径。
- 不写入用户尚未确认的路径。

## Harness 文件修改确认规则

主会话修改 harness/spec/plan/ledger/agent/hook 相关文件前，必须先判断本次修改属于哪一类。

### 必须先取得用户确认的修改

以下修改会改变协作结构、任务语义或执行边界，执行前必须向用户说明原因、影响文件和具体方案，并等待确认：

- 新建、迁移、重命名或删除 harness/spec/plan/ledger/agent/hook 文件或目录。
- 修改 `.codex-harness.toml`、hook 脚本、agent role 配置或 skill 规则。
- 修改 spec 的目标、需求、范围、非目标、约束、设计决策或验收标准。
- 修改 plan 的任务拆分、执行顺序、文件边界、验证策略、gate 条件或完成标准。
- 重写、压缩、删除 ledger 历史记录，或改变 ledger 的结构性组织方式。
- 任何可能让后续 agent 改变工作范围、实现方向、验证要求或权限边界的修改。

### 不需要逐次确认的状态更新

在用户已经确认某个任务进入 harness 执行后，主会话可以直接更新已确认的 plan/ledger，用于记录已经发生的事实或当前状态，例如：

- 勾选已完成的 plan checkbox。
- 更新 ledger 的 Current Snapshot / 当前状态。
- 记录子代理委派、结束、产出摘要和文件边界。
- 记录已运行的验证命令、结果、失败原因和阻塞原因。
- 记录风险、未决问题、阻塞状态的新增、解除或变化。
- 修正 ledger 中过期的“当前状态”，使其与最新事实一致。

### 边界规则

状态更新只能反映已经发生的事实，不得借状态更新改变需求、范围、验收标准、任务拆分、执行策略、文件边界、gate 条件、hook/agent/harness 配置或权限边界。

如果一次修改同时包含“事实状态更新”和“结构/语义变更”，必须拆开处理：事实状态可以直接记录；结构或语义变更必须先取得用户确认。

## 后续开发

项目 harness 明确后，再进入开发流程：

- 主会话负责对齐、计划、委派、复核和 ledger。
- 子代理负责被委派的调研、实现、测试或 review。
- 如果配置禁止主会话代码编辑，主会话不得接管任何代码或测试实现。
- 轻量任务继续保持轻量，不强制补全完整流程。
