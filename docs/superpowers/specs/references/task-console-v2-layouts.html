处置工作台第二版布局稿

本文件把第二版 UI/UX 讨论沉淀为静态画面。它不是实现代码，也不是 Stitch 生成结果；它用于约束后续设计和实现，避免工具根据长文字自由脑补。

核心模型：一个可左右吸附/收起的侧边协同面板、一个最大化业务卡片工作区、 一个默认隐藏的底部浮动处置指令坞。

数据源使用 police-command-workflow.json。同一个 task 的产出要同时对应三个位置： 业务卡片区展示可视化结果，处置动态展示 task 结果消息，预案进程展示 task 状态和是否有卡片。

v2.1 密度原则：顶部、tab、卡片标题、事件、task 行默认一行表达；无效 label 不进界面， 长原因进入详情或浮层，不挤占主屏高度。

  • 顶部栏：左侧是产品标识和标题，中间是当前警情名称与接警单号，右侧依次是运行状态、停止、重新执行、收起侧栏按钮。

控件说明：左侧吸附态

  • 吸附方向：同一个 side-panel 从右侧移动到左侧，数据、task 状态、卡片和消息都不改变。
  • 侧栏头部：左侧吸附时收起按钮方向变为向左，表示收起到左边缘 rail。
  • 协作 tab：仍由处置动态和预案进程组成，只改变视觉位置，不改变上下区域职责。
  • 卡片工作区：位于右侧，继续按剩余宽度自适应列数；最大化卡片只覆盖右侧卡片区域。
  • 适用场景：用户希望左手查看协作流、右侧大面积查看地图或监控时使用。

控件说明：队列 tab

  • tab 切换：协作、队列、历史是同一侧栏的三个模式，不再拆成多个侧边栏。
  • 队列列表：每条警情显示状态、进入时间、警情摘要和处置状态；当前处理项使用强调边框。

控件说明：指令坞 hover 浮出态

  • 触发方式：鼠标靠近浏览器底部热区时浮出；离开且输入框未聚焦时自动隐藏。
  • 常驻条件：点击输入框、选择失败 task、或点击侧栏头部命令按钮后，指令坞进入常驻状态。
  • 上下文 chip：显示当前警情或当前绑定 task；失败 task 场景下要显示具体 task 名称。
  • 工具按钮：＋ 用于上传附件，临用于临时协查，补用于补救 workflow，↑ 用于发送指令。
  • 数据流向：发送后的用户指令必须进入处置动态，并可能触发 main agent、workflow task 或 spawn agent。
  • 布局约束：指令坞是浮层，不占用 workspace 高度，也不改变卡片网格和侧栏高度。

控件说明：失败 task 选中态

  • 失败状态：顶部状态 pill 变为待补救，预案进程中失败 task 行选中并显示红色失败图标。
  • 三处产出：失败 task 不生成业务卡片，但必须在处置动态输出失败结果，并在预案进程保留失败状态。
  • 指令坞绑定：context chip 绑定“跟踪出警警车”，用户输入默认作为该 task 的补救指令。
  • 补救按钮：补用于启动补救 workflow，临用于启动临时协查，＋用于补充附件或设备信息。
  • 处置动态：用户补救指令、main agent 响应、后续 task 结果都追加到同一事件流中。
  • 高度策略：失败说明不应在 task 行里展开成长文本；长原因放在处置动态或点击详情里。
